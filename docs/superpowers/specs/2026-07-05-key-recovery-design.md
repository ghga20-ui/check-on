# 열쇠 복구·재발급 설계 (E2E 암호화)

작성일: 2026-07-05

## 배경 / 문제

E2E 암호화 도입 후 가장 큰 실사용 위험은 **열쇠 분실 = 데이터 영구 손실**이다.
열쇠는 데스크톱 PC(`%LOCALAPPDATA%/NeisSubject/sync_key.bin`, DPAPI)에만 있고,
PC 교체·초기화 시 드라이브의 암호화된 출결 기록을 다시 읽을 방법이 없다.
두 번째 위험은 **열쇠 유출 시 무효화 수단 부재**다(QR/복구 코드가 새면 되돌릴 수 없음).

## 원칙

- 열쇠의 **원천은 데스크톱**이다. 폰은 항상 수신자이며, 복구가 필요하면 데스크톱에서
  **재페어링**하면 된다. 따라서 복구·재발급 기능은 **데스크톱에만** 둔다(PWA 변경 없음).
- 복구 코드는 사람이 종이에 적었다가 몇 달 뒤 다시 입력하는 것을 전제로 설계한다.

## 복구 코드 형식

- 내용: `sync_key(32B) || SHA-256(sync_key)[:3]`(체크섬 3B) = 35바이트.
- 인코딩: **Crockford Base32**(대문자 알파벳 `0-9 A-Z`에서 I·L·O·U 제외) 비트스트림.
  35B = 280비트 = 정확히 56자(패딩 없음). 4자씩 끊어 하이픈으로 14묶음 표시.
  예: `XXXX-XXXX-…-XXXX`.
- 입력 관용성: 대소문자 무시, 하이픈·공백 제거, `O→0`, `I·L→1`로 교정.
- 검증: 길이(56자) → base32 디코드(35B) → 마지막 3B가 `SHA-256(앞 32B)[:3]`과 일치.
  불일치 시 오타로 보고 거부한다.
- 페어링 payload(`checkon.sync.v1:` + base64url)와는 별개다. 페어링은 QR 기계판독용,
  복구 코드는 사람 손입력용.

## 동작

### 복구 코드 보기/저장 (`get_recovery_code`)
- 현재 열쇠가 있으면 복구 코드를 반환, 없으면 null.
- **로그·커밋·클라우드에 남기지 않는다**(페어링 키와 동일 취급).

### 복원 (`restore_from_recovery_code`)
1. 코드 파싱 → 후보 열쇠.
2. **저장 전 검증**: 드라이브에 암호문 파일이 있으면, 그중 하나를 후보 열쇠로 복호화
   시도. 열리면 유효, 안 열리면 "이 코드로는 기존 데이터를 열 수 없습니다"로 거부(저장 안 함).
   암호문이 아직 없으면(신규) 그대로 수락.
3. 검증 통과 시 `save_sync_key` + 캐시 무효화.
- Drive I/O를 하므로 `SERIALIZED_API_METHODS`에 포함(httplib2 비-thread-safe).

### 재발급 (`reissue_sync_key`)
1. 새 열쇠 생성·저장 → 스토어 재생성(이후 쓰기는 새 열쇠로 암호화).
2. 드라이브 전체 순회: 각 파일을 **옛 열쇠로 복호화** → **새 열쇠로 재암호화**하여 upsert.
   평문 파일은 새 열쇠로 암호화. 파일별 best-effort `(reencrypted, failed)`.
3. 완료 후 옛 열쇠는 무효. **폰은 새 QR로 재페어링** 필요(안내 표시).
- 부분 실패 가능성(일부 옛키/일부 새키)을 UI가 `failed>0`로 노출하고 [다시 시도] 제공.
- Drive I/O + 다단계이므로 `SERIALIZED_API_METHODS`에 포함.

## 노출 (데스크톱 UI: 설정 → 기본 정보 → 모바일 연결 암호화 카드)

- **복구 코드 저장**: 코드 표시 + 복사 + "종이에 적어 안전한 곳에 보관" 안내.
- **복구 코드로 복원**: 입력란 + 복원 결과(성공/거부 사유).
- **열쇠 재발급**: 강한 경고("옛 열쇠 무효화 · 폰 재페어링 필요") + 진행 결과·재시도.
